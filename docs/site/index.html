Why go-code

Coding agents are useful. Losing track of them is not.

go-code is for engineers who want agent help without handing the whole repo to an opaque chat box. It keeps the work local, shows the moving parts, and gives you practical controls when the task changes.

Use it for

Failing tests

Ask it to reproduce a failure, inspect the relevant code, make a focused patch, and run the verification command.

Use it for

Codebase orientation

Drop into an unfamiliar repo and ask for the files, flows, and risks that matter before you start editing.

Use it for

Careful refactors

Keep a long task visible: tool calls, test output, costs, approvals, and the reasoning trail stay attached to the run.

Start in your terminal

Install it, then open it from the project you are already working in.

The current install path is source-based. It builds the local daemon and terminal client, then installs the go-code command and the assets it needs to run outside the checkout. 

git clone https://github.com/dennisonbertram/go-code.git
cd go-code
./scripts/install.sh --add-to-path

go-code                          # open the TUI
go-code "fix the flaky test"      # run once, then exit
go-code --server                 # use the HTTP runtime directly

The working loop

It behaves like a pair programmer you can interrupt.

Start with a plain request. As the run unfolds, you can see the tools, approve sensitive steps, add new context, cancel a bad direction, or continue the conversation later.

01

Launch inside a repo

Run go-code from the project root. The agent works against that workspace instead of a vague cloud copy.

02

Watch the work

See file reads, shell commands, patches, tool output, token usage, and costs while the run is still active.

03

Decide what happens next

Steer the task, approve or deny work, switch direction, continue a session, or stop the run when enough is enough.

Built for trust

The important parts stay inspectable.

Local first Run it from your machine against the repository you choose.
Visible tools File reads, edits, shell commands, approvals, and failures show up in the run stream.
Bounded runs Set model, provider, sandbox scope, approval policy, maximum steps, and cost limits per task.
Recoverable context Keep conversations, replay runs, compact long context, and continue without starting from scratch.

For tool builders

The same engine has a streamed HTTP API.

The terminal client is the easiest way in, but the runtime is scriptable. Start harnessd, create a run, and subscribe to Server-Sent Events for lifecycle state, tool activity, usage, approvals, summaries, todos, and conversation history.

POST /v1/runs GET /v1/runs/{id}/events POST /v1/runs/{id}/steer POST /v1/runs/{id}/approve GET /v1/models GET /v1/conversations/{id}/messages 
RUN_ID=$(
  curl -s http://127.0.0.1:8080/v1/runs \
    -H 'content-type: application/json' \
    -d '{
      "prompt": "Find the risky code path",
      "workspace_path": "'$PWD'",
      "model": "deepseek-v4-pro",
      "max_steps": 8
    }' | jq -r .run_id
)

curl -N "http://127.0.0.1:8080/v1/runs/$RUN_ID/events"
